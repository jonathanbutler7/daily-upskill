# Writeback

## Overview

When a payment has the `PAYMENT_SUCCEEDED` status, the writeback module will attempt a writeback to the PMS ledger using the required PMS ledger operations for the payment and records whether the result was full, partial, or failed.

## Terminal states

- `POSTED` the writeback posted to the ledger successfully
- `PARTIALLY_POSTED` writeback failed or was degraded to an unapplied insurance credit. Terminal status, does not trigger retries
- `WRITEBACK_FAILED` the writeback failed, and exhausted the retry policy

## Invariants
- The writeback module must only attempt writebacks on `payment_succeeded` records
- When the writeback module begins processing a payment it must move to `WRITING_BACK`
- A writeback attempt that fails must retry but with a limit of 3 and with an exponential backoff
- After retries are exhausted the state must transition to `WRITEBACK_FAILED`
- each writeback call must include an idempotency key, `external_reference_id`

## Idempotency

The PMS must treat identical `external_reference_id`s as idempotent. 

`external_reference_id` must be unique and generated by stable fields.

- payment posting: `payersync:{office_id}:{trace_number}:{claim_id}:payment`  
- adjustment posting: `payersync:{office_id}:{trace_number}:{claim_id}:cas:{index}`

## Ledger methods

At minimum the PMS ledger is guaranteed to provide the following operations:

- `post_insurance_payment(...)`
  - records actual dollars received from the payer and applied to a claim
- `post_adjustment(...)`
  - records reduction of the balance that is not collected cash
- `post_unapplied_insurance_credit(...)`
  - fallback method to be used when the payment is valid but can't be attributed to a claim
