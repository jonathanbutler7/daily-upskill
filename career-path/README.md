# Career Path to Director of Engineering

## Context

**Current Level**: Senior Engineer  
**Target**: Director of Engineering  
**Timeline**: ~5-7 years  
**Strategy**: Staff Engineer → EM → Senior EM → Director

## Why Staff First?

As a bootcamp grad, there's a perception gap around technical depth compared to CS-degree holders. Going Staff Engineer first addresses this directly:

1. **Credibility**: Staff promotion is definitive proof of technical competence
2. **Compensation**: Negotiate EM salary from Staff baseline, not Senior baseline
3. **Optionality**: Can return to IC track if management isn't the right fit
4. **Director foundation**: Directors need to evaluate Staff engineers and coach managers on technical decisions

## Current Gaps

- **System design and tradeoffs**: Need to move from feature-level thinking to system-level thinking
- **Cross-team visibility**: Currently scoped to single team; Staff requires org-wide impact
- **Technical authority**: Need to establish expertise in a specific domain

## The Plan

The 52-week plan in this directory is organized into 4 quarters:

| Quarter | Focus | Outcome |
|---------|-------|---------|
| Q1 | Technical Foundation | Deep understanding of 3 production systems |
| Q2 | Cross-Team Impact | Lead a multi-team technical initiative |
| Q3 | Technical Authority | Become the go-to person for a domain |
| Q4 | Promotion Case | Build evidence for Staff promotion |

## How I Learn Best

Based on my learning style profile:

- **Concrete-first**: See the problem before the solution. Start with broken code or a real scenario, then discover the theory.
- **Visual processing**: Code examples and diagrams beat abstract descriptions. Draw the architecture before reading about it.
- **One concept at a time**: Don't pile on multiple ideas. Master one thing before moving to the next.
- **Small experiments**: Predict outcomes before running code. Build intuition through hands-on practice.
- **Model-building**: Build high-level mental models first, then fill in details as needed.

### Applying This to the Plan

Each weekly goal should follow this pattern:
1. **Start with a concrete artifact** (diagram, code, doc) not abstract reading
2. **Ask "why" questions** to system owners - learn the rejected alternatives
3. **Write it down** - synthesizing into writing forces clarity
4. **Teach someone** - explaining to others reveals gaps in understanding

## Key Principles

- **Ship over study**: Designing and shipping systems beats reading about them
- **Artifacts over conversations**: Design docs, diagrams, and blog posts scale influence
- **Cross-team over single-team**: Staff is measured by org-wide impact
- **Tradeoffs over solutions**: Explain why it's the right tradeoff, not just what to build
