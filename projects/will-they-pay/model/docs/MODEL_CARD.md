# Model Card: Will They Pay? v1

## Overview

| | |
|---|---|
| **Model name** | `will_they_pay_prod_v1` |
| **Type** | Binary classifier |
| **Algorithm** | XGBoost with SMOTE resampling |
| **Artifact** | `models/will_they_pay_prod_v1.joblib` |
| **Version** | prod_v1 |
| **Last trained** | April 2025 |

Predicts whether a patient will pay an open invoice within 30 days. Output is used to surface a payment likelihood score and risk band in the Weave payments UI.

---

## Project Structure Standard

This project generally follows the Cookiecutter Data Science layout and conventions:
https://cookiecutter-data-science.drivendata.org/

**Aligned with that standard:**
- Split data directories by lifecycle (`data/raw`, `data/processed`, `data/output`, `data/external`)
- Separate model artifacts under `models/`
- Keep documentation under `docs/`
- Keep runnable data/feature/model workflows under `notebooks/` and `scripts/`

**Where this project intentionally deviates:**
- Notebook-first implementation for feature engineering and training (no dedicated `src/` Python package yet)
- Manual orchestration of the training/scoring flow (no workflow scheduler in this repo)
- CSV-backed prediction store for serving as an implementation placeholder; target production state is a Postgres table
- Local DVC setup for dataset versioning instead of a shared remote artifact store

---

## Intended Use

**Primary use:** Batch scoring of patients with open balances. Predictions are written to `data/output/scored_predictions_prod.csv` and served by the Go schema service via `GetPaymentLikelihood`.

**Intended users:** Weave payments product; front-desk staff acting on collection recommendations.

**Not intended for:**
- Real-time per-request scoring
- Medical, credit, or insurance decisions
- Scoring patients outside the Weave payments context

## Training Data

| | |
|---|---|
| **Source** | Production invoice and payment data snapshot (April 2025) |
| **Rows** | 24,351 invoices → 19,480 after filtering and feature joins |
| **Label** | `paid_within_30_days`: 1 if first successful payment arrived within 30 calendar days of invoice creation |
| **Class balance** | 78.4% paid (1), 21.6% unpaid (0); SMOTE applied to training fold to address imbalance |
| **Train/test split** | 80/20, stratified |

**Important caveat:** One of the 17 input features — `appointment_reliability_score` — is derived from synthetic calendar event data generated by `faker`, not from real appointment records. The feature ranks #3 in importance (12.6% gain) in the current model. This is a placeholder until the scheduler data source becomes available for training. See [Known Limitations](#known-limitations).

## Features

17 features, ordered as in `models/feature_columns_prod.json`.

| Feature | Category | Source |
|---|---|---|
| `amount` | Invoice | Real — `payments.invoices` |
| `created_day_of_week` | Invoice | Derived from invoice timestamp |
| `created_day_of_month` | Invoice | Derived from invoice timestamp |
| `created_hour` | Invoice | Derived from invoice timestamp |
| `payment_origin_mode` | Invoice | Real — mode of `payment_log.origin` per person |
| `surchargingenabled` | Invoice | Real — `payments.invoices` |
| `is_guardian` | Patient | Synthetic — `public.person` unavailable; dummy generated |
| `account_age_days` | Patient | Synthetic — derived from dummy `entry_date` |
| `tenure_months` | Patient | Synthetic — `account_age_days // 30` |
| `is_new_patient` | Patient | Derived — 1 if no prior payment history |
| `average_days_to_pay` | Behavior | Real — mean days from invoice creation to first payment |
| `appointment_reliability_score` | Behavior | **Synthetic** — ratio of attended to total appointments; dummy data |
| `median_household_income` | Census | Real — Census ACS5 2023, ZIP-level |
| `poverty_rate_pct` | Census | Real — Census ACS5 2023, ZIP-level |
| `average_household_size` | Census | Real — Census ACS5 2023, ZIP-level |
| `bachelors_or_higher_pct` | Census | Real — Census ACS5 2023, ZIP-level |
| `unemployment_rate_pct` | Census | Real — Census ACS5 2023, ZIP-level |

**PII policy:** No name, DOB, SSN, phone, email, raw dates tied to patient ID, or medical record numbers are used. Only opaque patient IDs, ZIP codes, derived aggregates, and census data.

## Evaluation

Evaluated on a held-out test set of 4,871 invoices.

| Metric | Value |
|---|---|
| **ROC-AUC** | **0.9469** |
| Brier score | 0.0778 |
| Training set paid rate | 78.4% |

Target thresholds from [success-criteria.md](ideation/success-criteria.md):

| Criterion | Target | Status |
|---|---|---|
| ROC-AUC | ≥ 0.70 (stretch 0.75) | ✅ Achieved 0.9469 |
| Precision on unlikely payers | ≥ 0.65 | See train_model_prod.ipynb Step 6 |
| Recall on risky accounts | ≥ 0.40 | See train_model_prod.ipynb Step 6 |

**Risk band thresholds:**

| Band | Confidence score | Interpretation |
|---|---|---|
| `high` | ≥ 0.70 | Likely to pay within 30 days |
| `medium` | 0.40 – 0.69 | Uncertain; monitor |
| `low` | < 0.40 | Unlikely to pay; prioritize outreach |

## Known Limitations

### `appointment_reliability_score` is synthetic

This feature is generated from fake appointment data and does not reflect real patient behavior. It carries 12.6% of the model's predictive weight. When scheduler data becomes available, this feature should be replaced and the model retrained. Until then, treat any prediction that is heavily influenced by this feature with caution.

### Patient and address data is synthetic

`is_guardian`, `account_age_days`, and `tenure_months` are derived from dummy records keyed to real patient IDs. `account_age_days` has a random offset applied, which affects both `tenure_months` and `is_new_patient`.

### Single data snapshot

The model was trained on one snapshot of invoice data (April 2025). It has not been tested for temporal drift. Retraining on a more recent snapshot is recommended before wider rollout.

### No automated retraining

Scoring and retraining are manual processes. Scores go stale as new invoices arrive. A nightly batch job is the planned production path (see [TPD.md](TPD.md)).

### No fairness audit

Disparate impact across patient ZIP codes or demographic groups has not been evaluated. ZIP-level census features are used as proxies; their effect on subgroup performance is unknown.

## Out of Scope

- Predicting payment at 60 or 90 days
- Online or real-time inference
- Automated financing or payment plan creation
- Drift detection or automated retraining
- Fairness audits (recommended before production rollout)
