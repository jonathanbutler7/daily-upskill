# Processor

## Overview

Once the reconciler finds a matched ERA and VCC payment group, the processor receives that and proceeds to execute one idempotent charge for the total matched amount. The processor is not responsible for matching or writebacks.

As soon as the processor finds a match in the DB or receives a match from the reconciler, it can execute the payment. 

This job does not have to run in sequence with the first two modules (ingester and reconciler), but must remain isolated so that it can run independent from (and possibly at a more frequent interval than) the first two modules

## Invariants

- The processor must only execute one charge per ERA / VCC payment group
- A payment must never reach the **`PAYMENT_SUCCEEDED`** state unless the processor receives a confirmation response from the stripe-like service
- If charging a VCC results in an error, the error must be reported and never swallowed

## Input Contract

The input for the processor is a matched ERA and VCC payment group. Matching logic is contained in the reconciler module and guarantees that the ERA and VCC payment group will have matching amounts. 

Example:
```go
type MinimalInputStruct struct {
    MatchedAmount          int64 // cents
    VccPaymentGroupID      string 
    TraceNumber            string 
    OfficeId               string 
    VccpaymentGroupVersion string
}
```

## Failure Boundary

The following failure scenarios must be handled and all result in `PROCESSING_FAILED`
- Network failure mid charge
- Payment method errors such as terminal decline
- Retry exhaustion

## Idempotency

An idempotency key must be generated by the processor module and passed into the stripe-like processor to prevent double charges in the case of network failure, or accidental retries.

Must be derived from stable identity fields:

Example:
- office_id + trace_number + matched_amount + vcc_payment_group_version
