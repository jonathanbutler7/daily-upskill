## 1. Posted Entries Balance

Question:

```text
How can Postgres prevent a posted ledger transaction whose entries do not sum to 0?
```

Think through:

```text
Can a CHECK constraint do this?
- Yes a CHECK can do this. A check a data validation rule that ensures all values in a column or group of columsn satisfy a condition before moving forward
Can a foreign key do this?
- A foreign key doesn't have any bearing on preventing bad transactions, unless the key is used to join a table that has data needed for a CHECK.
Does this require a trigger or stored procedure?
- Could be overengineering unless there a postgres functions that need to fire to update other tables based on conditions
Should Go calculate it first too?
- Hm, could go either way. It's reasonable to trust that postgres validations will work. and only have validation at one stage. however, since this is a financial db, it could also make sense to have an extra layer of validation before the postgres validations are reached
What happens if entries are inserted one at a time?
- it will be possible for ledger entries to get out of date from one another. entries should be grouped into a single db transaction.
```

```text

```

Think through:

```text

- idempotency kek should be stored on the transaction and on the entry
- idempotency key should be generated by application code using stable unique fields
What should be unique?
- idempotency key should be a one way generated key based on stable fields in the entry and transaction records. an example could be transction id + amount + personid or something

- this is exactly the use case for idempotency keys. handle them correctly by makin gsure they both return the same result.
```

## 4. No Negative Balance

Question:

```text
How can the system prevent Alice from spending more than her available balance?
```

Think through:

```text
Do I need to read Alice's balance before posting?
- yes we need to read alice's balance in the COMMIT; before doing anything. we can also read it in theapplication layer to have a second layer of validation. it's helpful to have the double check in two layers in case a bug ever makes its way into the system, we have a fallback
What happens if two requests read the same balance?
- maybe we use a PG thing here to make sure the 2 requests reading the same balance doesn't happen due to a row lock
Should I lock Alice's account row?
- depends if account has balance information. we'd need to lock so that we can update the amount, then release the row, and ensure no other transactions read an out of date balance and thus send a transaction they shouldn't
Should I use SERIALIZABLE and retry failures?
- yes
```
